# 🏠 Real Estate Chatbot — RAG-Powered Property Assistant

A retrieval-augmented chatbot that answers questions about a real estate listings catalogue — property details, amenities, inspection reports, FAQs — using only grounded, verifiable data instead of LLM guesswork. Built as part of my AI Internship at Prudent Infotech, where I benchmarked production LLM options (GPT-4.1 API vs. open-source/self-hosted alternatives) and shipped this as a fully local, privacy-friendly reference implementation running on Llama 3.1 via Ollama.

---

## ✨ What it does

Ask it things like:

- *"Show me all 2BHK in Pune"* → enumerates every matching listing from the structured table
- *"Homes with a swimming pool under 2 crore"* → filters on price + amenities together
- *"What did the inspection report say about the Powai flat?"* → semantic search over unstructured documents (brochures, inspection notes, FAQs)
- *"Is pet allowed in the Banjara Hills apartment?"* → pulls the exact FAQ passage for that listing

The bot never invents prices, fees, or amenities — it's instructed to answer strictly from retrieved context and say so plainly when information isn't available, citing which listing each answer comes from.

---

## 🧠 How it works

The system runs in two phases:

**Phase A — Ingest.** Raw listing data (CSV/JSON, including free-text brochures, inspection reports, and FAQs) is parsed, normalized into structured columns, and chunked into searchable passages. Each chunk is embedded with a sentence-transformer model and stored alongside its metadata.

**Phase B — Retrieve → Prompt → Generate.** Every incoming question is classified into one of two retrieval modes before an LLM ever sees it:

| Mode | Triggered by | Behaviour |
|---|---|---|
| **List** | Catalogue-style queries ("show me", "all", "which flats", amenity/feature mentions) | Runs a structured SQL query with extracted filters (city, bedrooms, price ceiling, amenities) and returns **every** matching listing — not just the top few, which is what causes a bot to "only mention 4 properties" when there are 12. |
| **Semantic** | Specific, detail-level questions about a particular property or document | Embeds the question and runs a vector similarity search (k-nearest-neighbours) over chunked brochures/inspections/FAQs, then narrows results with any extracted filters (listing, city, bedrooms, price). |

Retrieved chunks are assembled into a grounded prompt with strict system instructions (answer only from context, attribute claims to the right listing, never guess), then sent to a locally-running Llama 3.1 model through Ollama's OpenAI-compatible API — making it a drop-in swap to a hosted API later by just changing a base URL and key.

## 🧱 Database Design

Built on **SQLite** extended with **sqlite-vec** for native vector search — no separate vector database needed.

- **`listings`** — one row per property, with typed columns for everything you'd filter on: price, bedrooms, bathrooms, area, floor, facing, parking, furnishing, year built, maintenance fee, locality, city, and status, plus a generated natural-language `property_card` summary used for list-mode display.
- **`listing_features`** — a flexible key/value table for anything that doesn't fit a fixed column (pet policy, water supply, possession date, RERA ID, etc.), so the schema doesn't need to change every time the data has a new attribute.
- **`chunks`** — every searchable passage (property card, description, brochure, inspection report, FAQ, or general/global doc) with its source and listing reference.
- **`chunk_vectors`** — a `vec0` virtual table storing 384-dimensional embeddings (from `BAAI/bge-small-en-v1.5`) for fast approximate nearest-neighbour search over `chunks`.

## ⚙️ Tech Stack

- **Python 3** — core application logic
- **SQLite + sqlite-vec** — combined structured store and vector index in a single file
- **sentence-transformers (`bge-small-en-v1.5`)** — text embeddings for semantic search
- **Ollama + Llama 3.1 (8B)** — local LLM inference via an OpenAI-compatible endpoint
- **openai (Python SDK)** — used purely as a client against Ollama's compatible API, so swapping to a hosted provider later is a one-line change

## 🚀 Getting Started

**1. Install dependencies**
```bash
pip install openai sentence-transformers sqlite-vec
```

**2. Install Ollama and pull a model**
```bash
# https://ollama.com
ollama pull llama3.1:8b      # if you have an 8GB+ GPU
ollama pull llama3.2:3b      # smaller / lower-VRAM fallback
```

**3. Build the knowledge base**
```bash
python ingest.py
```
This reads everything in `data/` (CSV + JSON listings, including nested brochure/inspection/FAQ documents) and populates `realestate.db` with structured rows, text chunks, and embeddings.

**4. Chat with it**
```bash
python chat.py
```
```
Real-estate assistant. Ask a question (Ctrl+C to quit).

You: show me 3bhk under 2 crore in thane
Bot: ...
(sources: listing 903)
```

You can also test retrieval directly without the LLM in the loop:
```bash
python retrieve.py "show me all 2bhk in pune"
```

## 🔍 Example Filters the Retriever Understands

- Bedrooms — *"2BHK"*, *"3 bed"*
- Price ceilings — *"under 1.5 crore"*, *"below 80 lakh"*
- City / locality — matched directly against the listings table
- Amenities — pool, gym, garden, parking, clubhouse, pet-friendly, and more
- Status — available / sold / pending
